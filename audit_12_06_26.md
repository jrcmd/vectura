# Audit Vectura — 12/06/2026

## Résumé rapide
Le projet Vectura est un MVP bien structuré : backend Express/TypeScript avec Prisma, frontend React/Tailwind, PostgreSQL et Docker. La plateforme couvre les besoins essentiels de matching, validation documentaire, facturation et back-office. Les principaux risques tiennent à une architecture monolithique encore peu optimisée pour le trafic, des protections de sécurité incomplètes et des opérations de backup/monitoring encore basiques.

> Nota : l’analyse se base sur le code disponible dans `backend/`, `frontend/` et les fichiers de configuration. Aucun usage effectif de Redis n’a été trouvé dans le code backend, malgré la présence du service dans `docker-compose.yml`.

---

## AXE 1 — ARCHITECTURE & SCALABILITÉ
Score : 6.5/10

### Forces
- Architecture monolithique cohérente pour un MVP : backend unique + frontend séparé.
- `docker-compose.yml` et `docker-compose.prod.yml` sont déjà configurés pour le déploiement et la persistance.
- Matching et distance sont implémentés de façon claire dans `backend/src/services/matchingService.ts` et `backend/src/services/distanceService.ts`.

### Faiblesses & Risques
- Critique : le matching utilise un chargement de toutes les missions ouvertes en mémoire (`findMany` sans pagination ni filtre spatial) → risque de montée en charge.
- Haute : plusieurs requêtes en boucle dans `findCompatibleMissions()` pour vérifier les favoris, créant un comportement N+1.
- Normale : absence de configuration explicite du pool PostgreSQL dans Prisma ; laisse la gestion à la configuration par défaut.
- Normale : Redis est présent en infra mais non utilisé dans le backend visible, ce qui indique un potentiel blocage ou dette technique.

### Recommandations
- CRITIQUE : transformer le matching en requêtes filtrées côté DB avec index spatial approximatif ou pré-filtre sur statut/date/zone. Priorité à `backend/src/services/matchingService.ts`.
- HAUTE : regrouper les vérifications de favoris en un seul `findMany` plutôt qu’un `findFirst` par mission.
- HAUTE : activer et documenter l’usage de Redis pour cache de matching ou de sessions si nécessaire, ou retirer le service si inutile.
- NORMALE : configurer les paramètres de pool Prisma via `DATABASE_URL` et/ou `datasource.db.pool` si besoin.

### Checklist de vérification
- [ ] Tester les performances de `GET /api/driver/missions` ou équivalent avec 1000+ missions.
- [ ] Vérifier si Redis est attendu pour une fonctionnalité non implémentée.
- [ ] Contrôler le nombre de connexions PostgreSQL en charge via `pg_stat_activity`.
- [ ] Métriques de latence sur `matchingService.findCompatibleMissions()`.

## AXE 2 — SÉCURITÉ & CONFORMITÉ
Score : 6/10

### Forces
- Authentification JWT avec access token court (`15m`) et refresh token persistant en base.
- RBAC explicite avec `requireAdmin`, `requireCompany`, `requireDriver`.
- Uploads documentaires limités en extension et taille via Multer.

### Faiblesses & Risques
- Critique : fallback `process.env.JWT_SECRET ?? 'change-me'` ouvre un risque si la variable est absente en dev/test.
- Haute : pas de protection CSRF explicite et CORS origin `true` autorise potentiellement des origines indésirables.
- Haute : l’upload sécurise uniquement l’extension, pas le type MIME réel ni le scan antivirus.
- Haute : pas de rate limiting détecté dans le code malgré la présence de `rate-limiter-flexible` dans les dépendances.
- Normale : gestion des mots de passe avec SHA-256 salé (custom) est moins robuste que bcrypt/argon2.
- Normale : aucun mécanisme explicite de suppression RGPD ou droit à l’oubli dans les routes.

### Recommandations
- CRITIQUE : refuser le démarrage si `JWT_SECRET` n’est pas défini, plutôt que d’utiliser `change-me`.
- HAUTE : activer un middleware de rate limiting sur les routes sensibles (`/api/auth`, `/api/documents/upload`, `/api/companies/login`).
- HAUTE : limiter strictement `cors.origin` aux domaines connus en production.
- HAUTE : ajouter un contrôle MIME et, si possible, un scan de fichiers sur upload dans `backend/src/routes/documentsRoutes.ts`.
- NORMALE : faire évoluer le hash de mot de passe vers bcrypt/argon2 sur les nouveaux comptes, ou documenter strictement la compatibilité.
- NORMALE : définir une politique RGPD claire et auditor les routes de suppression/archivage.

### Checklist de vérification
- [ ] Vérifier l’existence de `JWT_SECRET` sur staging/production.
- [ ] Simuler attaque CSRF sur endpoints authentifiés.
- [ ] Tenter upload de fichier malveillant `.pdf` contenant un script.
- [ ] Revue ciblée des routes `requireCompany`, `requireAdmin`, `requireDriver`.
- [ ] Vérifier l’implémentation de la révocation du refresh token sur logout.

## AXE 3 — QUALITÉ DU CODE & MAINTENABILITÉ
Score : 7/10

### Forces
- TypeScript utilisé sur backend et frontend avec `eslint` et `jest` déjà présents.
- Architecture de dossiers claire : `backend/src/routes`, `services`, `middleware`, `lib`.
- Documentation de projet solide dans `README.md` et `DOCUMENTATION/`.

### Faiblesses & Risques
- Haute : duplication de logique JWT dans `requireAdmin.ts`, `requireCompany.ts`, `requireDriver` (`authRoutes.ts`).
- Normale : l’usage de middleware d’authentification n’est pas centralisé dans un utilitaire commun.
- Normale : les tests sont présents mais la couverture globale n’est pas évidente; des routes métier critiques (matching, sanctions, facturation) peuvent manquer.
- Normale : dépendance `rate-limiter-flexible` installée sans usage visible, ce qui alourdit le projet.

### Recommandations
- HAUTE : extraire la vérification JWT/autorisation dans un middleware partagé.
- HAUTE : ajouter des tests d’intégration pour `matchingService`, `documentsRoutes`, `missionService` et `billingService`.
- NORMALE : vérifier la couverture ESLint/TS sur l’ensemble du backend et corriger les anti-patterns.
- NORMALE : supprimer les dépendances inutilisées ou documenter leur usage attendu.

### Checklist de vérification
- [ ] Exécuter `npm run lint` et corriger les erreurs restantes.
- [ ] Lancer `npm test` et analyser la couverture.
- [ ] Rechercher les doublons de logique d’authentification.
- [ ] Vérifier la complétude des tests dans `backend/src/__tests__/`.

## AXE 4 — BASE DE DONNÉES & PERFORMANCES
Score : 6.5/10

### Forces
- Schéma Prisma globalement normalisé et adapté au MVP.
- Bons modèles métier : `Document`, `Mission`, `Favorite`, `Sanction`, `Billing`, `Invoice`, `AuditEvent`.
- Indices présents pour les logs d’audit, incidents et tâches cron.

### Faiblesses & Risques
- Haute : absence d’indices clairs sur les tables métier principales (`Mission.status`, `Mission.missionDate`, `Mission.creatorId`, `Document.expiryDate`, `Favorite.companyId`/`driverId`).
- Haute : requêtes potentiellement lentes dans `findCompatibleMissions()` et `missionReminderScheduler`.
- Normale : pas d’indication d’un rollback de migration ; Prisma migrate est utilisé mais sans procédure de retour.
- Normale : stratégie de sauvegarde locale via `pg_dump` stockée dans le conteneur, ce qui n’est pas sûr en production.
- Normale : pas de chiffrement des données sensibles au repos (ni pour les uploads, ni pour les dumps).

### Recommandations
- CRITIQUE : ajouter des index sur `Mission(status, missionDate)`, `Document(expiryDate)`, `Favorite(companyId, driverId)`.
- HAUTE : modifier le matching pour limiter et filtrer côté DB, réduire la charge et éviter le chargement de l’ensemble des missions.
- HAUTE : documenter la procédure de rollback Prisma et tester une restauration sur un environnement de staging.
- NORMALE : externaliser les sauvegardes `BACKUP_DIR` hors du conteneur et chiffrer les dumps si nécessaire.
- NORMALE : ajouter une stratégie pour supprimer ou anonymiser les données personnelles anciennes.

### Checklist de vérification
- [ ] Réviser les indices de `prisma/schema.prisma` pour les requêtes métier.
- [ ] Exécuter EXPLAIN sur les requêtes `findMany` les plus lourdes.
- [ ] Valider le process `npm run prisma:migrate` dans un environnement de staging.
- [ ] Valider la restauration d’un dump PostgreSQL depuis `backend/src/services/backupService.ts`.

## AXE 5 — FONCTIONNALITÉS MÉTIER & LOGIQUE MÉTIER
Score : 7/10

### Forces
- Matching cohérent autour du statut, qualifications et distance.
- Logique de priorisation favoris implémentée dans `matchingService`: le conducteur favori a accès pendant la fenêtre prioritaire.
- Sanctions et relances documentaires automatiques présentes via cron jobs.

### Faiblesses & Risques
- Haute : logique de discipline complète (3e désistement, suspension 7j) n’est pas repérée de façon explicite dans les fichiers inspectés.
- Normale : tarification plancher mentionnée en config (`MIN_RATE_PL`, `MIN_RATE_SPL`) mais non inspectée directement dans le code extrait.
- Normale : le cycle de vie des missions n’est pas fortement verrouillé par une machine d’états ; plusieurs transitions semblent possibles sans vérification centralisée.
- Normale : conformité URSSAF dépend d’un mail de relance, mais le blocage automatique n’est pas clairement visible.

### Recommandations
- HAUTE : documenter et vérifier les règles exactes de sanction / radiation dans `missionService` et `lateCancellationScheduler`.
- HAUTE : auditer les chemins de création de mission pour s’assurer que les tarifs PL/SPL sont vérifiés côté backend.
- NORMALE : transformer la gestion du cycle de vie en fonctions métier réutilisables plutôt qu’en simples mises à jour de statut.
- NORMALE : valider que l’attestation URSSAF expirée bloque bien l’attribution de nouvelles missions.

### Checklist de vérification
- [ ] Vérifier les scénarios de désistement dans `backend/src/services/missionService.ts`.
- [ ] Vérifier les règles tarifaires dans les routes de création/modification de mission.
- [ ] Tester le comportement favoris pendant la fenêtre prioritaire.
- [ ] Vérifier l’exactitude des notifications URSSAF et le blocage associé.

## AXE 7 — INFRASTRUCTURE & OPS
Score : 7/10

### Forces
- Dockerfile backend avec `HEALTHCHECK` et build propre.
- `docker-compose.yml` et `docker-compose.prod.yml` couvrent frontend, backend, PostgreSQL et Redis.
- Monitoring et audit implémentés via `AuditEvent`, `CronJob`, `IncidentTicket` et pages admin.

### Faiblesses & Risques
- Haute : conteneur backend s’exécute en root, pas de `USER` non-root.
- Haute : sauvegardes locales en volume (`pg_dump`) ne suffisent pas pour la production sans stratégie externe et chiffrement.
- Normale : absence de stock d’alertes formelles, alerting dépend seulement de logs console et tables de monitoring.
- Normale : cron jobs s’exécutent dans le même processus backend, ce qui limite la résilience en cluster.

### Recommandations
- HAUTE : exécuter l’image backend avec un utilisateur non-root.
- HAUTE : externaliser les sauvegardes vers un stockage durable hors conteneur (S3/NFS) et chiffrer les dumps.
- HAUTE : garantir que les jobs cron ne sont lancés qu’une seule fois en production ou les déplacer vers une instance dédiée.
- NORMALE : ajouter un outil d’alerting simple (e-mail/Slack) pour `incident` et échec de cron job.

### Checklist de vérification
- [ ] Vérifier que `HEALTHCHECK` remonte bien l’état API.
- [ ] Contrôler les volumes `prod_uploads` et `prod_backups` en production.
- [ ] Vérifier l’exécution de `prisma migrate deploy` au démarrage.
- [ ] Valider la cohérence des pages monitoring dans l’admin.

## AXE 8 — NOTIFICATIONS & AUTOMATISATION
Score : 7.5/10

### Forces
- Service mail avec fallback Ethereal et retries dans `backend/src/services/mailService.ts`.
- Jobs planifiés pour documents expirants, rappels mission J-1/J0 et sanction expiry.
- Journalisation basique des notifications via `NotificationLog`.

### Faiblesses & Risques
- Haute : absence de dead letter queue ou de file d’attente dédiée en cas de surcharge SMTP.
- Haute : erreurs d’envoi sont consignées au niveau console, pas de système de réessai automatique centralisé.
- Normale : `NotificationLog` est utilisé, mais le suivi de l’échec / reprogrammation n’est pas robuste.
- Normale : la logique d’envoi J-30/J-7/J-1 est embarquée dans une seule tâche, donc difficile à opérer séparément.

### Recommandations
- HAUTE : ajouter un mécanisme de retry plus robuste et un statut d’échec permanent dans `NotificationLog`.
- HAUTE : externaliser les notifications critiques dans une queue ou un service dédié si le trafic monte.
- NORMALE : enrichir les logs de notification avec des métadonnées d’erreur et des timestamps plus précis.
- NORMALE : tester la délivrabilité SMTP dans staging avec un vrai fournisseur.

### Checklist de vérification
- [ ] Simuler l’échec SMTP et vérifier l’état de `NotificationLog`.
- [ ] Vérifier que J-1/J0 sont bien déclenchés et marqués.
- [ ] Contrôler la création des notifications `DOC_EXPIRY` et `DOC_EXPIRED`.
- [ ] Vérifier la résilience du service mail au redémarrage.
